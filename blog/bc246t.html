BC246T — A Serial Protocol Library for a Radio Scanner

2026-06-12

The Uniden BC246T is a handheld trunking scanner — it listens to police, fire, aircraft, and other radio traffic across a wide frequency range. Programming it by hand is tedious: scrolling through menus, entering frequencies one at a time, setting modulation and squelch codes. For anyone with more than a handful of channels, it's unbearable.

So I reverse-engineered the serial protocol and wrote a Python library to control the thing from a computer.

The Protocol

The BC246T speaks a simple ASCII serial protocol at up to 57600 baud. Every command is a three-letter mnemonic followed by comma-separated arguments, terminated with \r. The scanner responds with the command echo, comma-separated values, and an OK/NG/ERR status.

PC → Radio:  MDL\r
Radio → PC:  MDL,BC246T

PC → Radio:  STS\r
Radio → PC:  STS,[16 chars display],[16 chars mode],...

The library wraps every command as a Python method or property. model returns "BC246T". firmware returns something like "1.00". status returns a dict with the display lines, icon states, squelch status, battery level, and weather alert.

You can read the scanner's current state, send keypresses, tune to arbitrary frequencies via Quick Search (VFO mode), and even power it off remotely.

Programming Mode

The real power is in Program Mode. Enter it, and you can read and write every setting the scanner has — systems, channels, groups, quick keys, lockouts, delays, attenuation, alert behaviors.

The library exposes this as two main objects:

Settings — Backlight timeout, battery save, key beep, opening message (the two-line text shown at power-on), priority mode. Getting and setting is just property access:

dev.settings.backlight = dev.settings.BACKLIGHT.INFINITE
dev.settings.opening_message = ("FREQ SCAN", "   ONLINE   ")

Systems — A list-like object that represents the scanner's system database. Systems have types (Conventional, Motorola Type II, EDACS, LTR, etc.), names, quick keys, hold times, and lockout states. Each system contains channel groups, which contain channels.

sys = Device.System(dev, sys_type=Device.System.SYS_TYPE.CONVENTIONAL)
sys.name = "AIR BAND"
sys.quick_key = 1
dev.systems.append(sys)

The entire serial protocol spec is documented inline in the source code — every command, every response field, every edge case. I transcribed it from Uniden's reference docs so the code served as both library and documentation.

The AVR Autoprogrammer

The same repo contains a companion hardware project: an AVR microcontroller that reads scanner programming data from an SD card over SPI, using a FAT16 filesystem. The idea was to create a standalone device — plug in an SD card with your channel config, connect the AVR to the scanner's serial port, and it programs everything automatically. No computer needed.

It initializes the USART, SPI, SD card, and FAT16 filesystem, then would read configuration files and send the appropriate serial commands to the scanner. The code is in C, targeting an 8 MHz AVR, with custom SPI and SD card drivers written from scratch.

Abandoned, But Complete

Unlike the uTorrent Bridge which was mostly stubs, the BC246T library is genuinely useful. The protocol implementation is thorough, the API is clean, and the documentation is excellent. I used it to program my scanner more than once.

The AVR autoprogrammer never got past initialization and filesystem mounting — the FAT16 driver was written but the actual scanner programming logic was never implemented. Still, getting raw SPI talking to an SD card on a bare-metal AVR is a rite of passage, and this code captures that moment.

It's Python 2, it uses map and lambda freely, and the tuple-unpacking syntax (def opening_message(self, (l1_char, l2_char))) is a fossil from an older Python. But it works. And 10+ years later, it's still a solid example of what you can do when you just read the datasheet and start writing code.

What I Learned

  • Documentation as code — Embedding the full protocol spec in the source comments meant the documentation and the implementation never drifted apart. It made maintenance trivial and debugging self-contained.
  • CVT (computer-verified testing) — Having a real device to test against made the difference between a library that works and one that almost works. Every command I implemented I could test immediately by watching the scanner's display respond.
  • Bare-metal AVR is a different world — Writing SPI and SD card drivers from scratch on an 8-bit microcontroller teaches you things about hardware timing and bit-banging that no high-level framework ever will.
  • Sometimes Python 2 was fine — The code still works a decade later. The right tool is the one that gets the job done, not necessarily the newest one.
